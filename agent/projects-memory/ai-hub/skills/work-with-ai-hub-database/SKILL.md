## When to Use
Use when working with AI Hub database — adding tables, modifying schema, creating migrations, or understanding the data model. Triggers on: database, migration, schema, Drizzle, table, column, pgvector, PostgreSQL.

## Procedure
1. Identify the DB change: new table, column addition, index, or relation.
2. Modify schema in src/server/infrastructure/db/schema-tables/{feature}.ts.
3. Export the table from src/server/infrastructure/db/schema-tables.ts.
4. Add relations in src/server/infrastructure/db/schema-relations.ts if the table has FKs.
5. Run npm run db:generate to create migration files.
6. Review the generated SQL in src/server/infrastructure/db/migrations/.
7. Run npm run db:push to apply locally (dev only) or npm run db:migrate for production.
8. Update docs/database-reference.md with the new schema.
9. Update docs/architecture.md if the change affects module architecture.

## Pitfalls
- Don't write migration SQL manually — use drizzle-kit generate.
- Always add FK constraints with proper cascade behavior.
- Use consistent column naming: snake_case for DB, camelCase for TypeScript.
- Always add createdAt/updatedAt timestamps to new tables.
- UUID columns should use .defaultRandom() for primary keys.
- Text columns with length limits use varchar('name', { length: N }).
- JSON columns use jsonb() for PostgreSQL JSON support.
- Update schema-relations.ts when adding tables with foreign keys.
- Check existing migration count to avoid conflicts.

## Verification
1. Schema file exists and exports the table correctly.
2. Table is exported from schema-tables.ts barrel file.
3. Relations are added in schema-relations.ts (if applicable).
4. Migration file is generated by drizzle-kit.
5. Migration SQL looks correct (FKs, indexes, defaults).
6. npm run typecheck passes with the new schema.
7. docs/database-reference.md is updated.
